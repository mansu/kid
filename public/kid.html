Kid Pages

Motivation

Man pages have been the staple of unix documentation since 1971. While a troff formatted page with an alphabetically sorted list of options with optional background is good enough for seasoned sysadmins and engineers, they are not so great for more application level programmers, generalists and casual users.

Man pages have several issues.

  • Alphabetical list of options: A man page is an alphabetical list of options. Options are not even grouped based on functionality.
  • Lack of examples: A man page rarely contains examples or spells out most common uses of the command.
  • The meaning of the output: Since man pages only written to explain the command line options, they don't do a good job of explaining the output of the program.
  • Back ground: Man pages don't provide the necessary background to understand the command. For example, it will be great if a man page for top provided some info about the linux memory model to help me understand the output of the top output.
  • Implementation details: Given that command line tools are heavily relied upon for one off programming tasks, it would be useful to know their performance characteristics or implementation details.

Kid Pages

Unlike man pages which only provide a listing of input and output options with some limited context, kid pages aim to provide more context sensitive documentation with examples, with more depth and breadth than a man page.

In a normal text doument, that covers a lot of depth and breadth, a lot of information would be irrelavent most of the time to most of the users. Hence, we use a new non-linear documentation format called docbubbles that allows us to write non-linear textual documentation for writing kid pages.
